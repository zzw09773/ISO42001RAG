B · Pre-flight：先確認上游服務可用

在 內網部署機器 上跑，不需要先傳檔案。任何一步失敗都先別繼續。

B-1 · LLM 端點可達 + 列出可用模型

內網
curl -s http://10.53.100.12:7000/v1/models | python3 -m json.tool
預期輸出（找到 gpt-oss-20b 才算過）
{
    "data": [
        {
            "id": "gpt-oss-20b",
            ...
        }
    ]
}
若失敗
  • connection refused → vLLM 沒起，去 LLM 主機 systemctl status vllm
  • 404 / 模型清單沒有 gpt-oss-20b → 名稱對不上，記下實際 id（如 openai/gpt-oss-20b），稍後 .env 用實際名

B-2 · gpt-oss-20b 真的會回覆（含 reasoning_effort 驗證）

內網
curl -s http://10.53.100.12:7000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{
    "model":"gpt-oss-20b",
    "messages":[{"role":"user","content":"回覆 OK"}],
    "max_tokens":50,
    "reasoning_effort":"medium"
  }' | python3 -m json.tool
預期
"choices": [{
  "message": {
    "role": "assistant",
    "content": "OK",        ← 非空字串才算過
    "reasoning": "User says: ..."
  },
  "finish_reason": "stop"
}]
若 content 是空字串或單一字元
代表 reasoning_effort 沒生效，重跑加大到 "reasoning_effort":"high" 看是否能回。內網 vLLM 版本若不支援這個參數，必須升級 vLLM 否則 rerank 一定壞（單字元被當「全拒絕」）。

內網
curl -s -X POST http://10.53.100.12:9000/v2/repository/index -d '{}' | python3 -m json.tool
預期（看到 nv-embed-v2 + READY）
[
  {"name": "nv-embed-v2", "version": "1", "state": "READY"},

C · 9000 vs 9001 — Triton 埠口釐清

你問該連哪個。答案是：embed-proxy 連 9001 (gRPC)，不是 9000 (HTTP)。Triton 一個服務開兩個埠：

Port協定用途誰連
9000HTTP/REST查模型清單、debug、健康檢查你用 curl 驗證（§B-3）
9001gRPC實際 inference（embedding 計算）embed-proxy 走這個
9002HTTPPrometheus metricsmetrics 用，本部署可忽略

驗證 9001 確實在跑（gRPC 不吃 curl，所以只能用 TCP probe 確認 port open）：

內網
nc -zv 10.53.100.12 9001 2>&1 | tee /tmp/triton-grpc.log
預期
Connection to 10.53.100.12 9001 port [tcp/*] succeeded!
為什麼不是 9000
embed-proxy 用的是 tritonclient.grpc.InferenceServerClient，只說 gRPC。連到 9000 (HTTP) 會丟 protobuf decode error → embed-proxy 起不來 → rag-api healthcheck fail → 整串卡住。

F · 內網：備份 + 解開 + .env 設定

F-1 · 先備份現有部署（一定要做）

內網
cd ~/ISO42001Deploy
TS=$(date +%Y%m%d_%H%M%S)
tar --exclude='RAG/data' \
    -czf ~/iso42001-backup-${TS}.tar.gz RAG docker-compose.yaml .env
echo "備份完成: ~/iso42001-backup-${TS}.tar.gz"
注意
備份排除 data/，因為那是要保留的，不需要備份。

F-2 · 解開新 code（覆蓋 .py / docs / docker-compose）

內網
cd ~/ISO42001Deploy
unzip -o ~/iso42001-patch/full-stack-code.zip

-o 是 overwrite without prompt。因為 zip 內已排除 data/，內網實際 audit_logs / versions 不會被動。

F-3 · diff 對照 docker-compose.yaml

內網
diff <(tar xzf ~/iso42001-backup-${TS}.tar.gz docker-compose.yaml -O) docker-compose.yaml
本 session 沒動 docker-compose.yaml
這個 diff 預期應該是空的。如果有差異而你不是故意改的，cp ~/iso42001-backup-${TS}.tar.gz 解回去舊版。

F-4 · 編輯 .env — 這是內網最關鍵的設定

內網
nano .env

確認以下欄位（沒有就加）：

# === 上游服務（內網直連，無 https）===
LLM_HOST=10.53.100.12

LLM_API_BASE=http://10.53.100.12:7000/v1
LLM_API_KEY=             # vLLM 通常不檢查；留空或設任意字串
CHAT_MODEL_NAME=gpt-oss-20b

# === Embed：走本地 embed-proxy，不直連 Triton ===
EMBED_API_BASE=http://embed-proxy:7100/v1
EMBED_API_KEY=           # embed-proxy 不檢查 key
EMBED_MODEL_NAME=nv-embed-v2

# === embed-proxy 連 Triton 的 gRPC 埠（重要！9001 不是 9000）===
EMBED_GRPC_PORT=9001
TRITON_MODEL_NAME=nv-embed-v2

# === Reasoning（gpt-oss-20b 必須的）===
REASONING_EFFORT=medium

# === SSL（內網 http，所以開）===
VERIFY_SSL=true

# === 內網模式 + 信任的 proxy ===
ALLOW_INTRANET_MODE=true
TRUSTED_PROXIES=127.0.0.1,nginx

# === 主機 UID/GID（deploy.sh 會自動覆蓋）===
HOST_UID=1000
HOST_GID=1000
三個常踩的雷
  • CHAT_MODEL_NAME 要跟 §B-1 看到的 id 完全一致（有些 vLLM 會帶 openai/ 前綴）
  • EMBED_GRPC_PORT=9001，預設值 9000 是錯的
  • EMBED_API_BASE 不要設成 http://10.53.100.12:9000/v1，要走 embed-proxy:7100。直連 Triton HTTP 我們的 OpenAI client 不會講話。

H · 部署後驗證（4 件事）

H-1 · rag-api 看得到模型 + 認證模式正常

內網
curl -s http://localhost:8043/v1/models | python3 -m json.tool
預期
應該回 rag-agent 一筆。

H-2 · 端對端 query — 看 retrieve 不再 empty

內網
curl -s http://localhost:8043/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -H 'x-session-id: deploy-check' \
  -d '{
    "model":"rag-agent",
    "stream":false,
    "messages":[{"role":"user","content":"陸海空軍懲罰法第5條規定什麼？"}]
  }' | python3 -m json.tool | head -20
預期
content 非空字串、有提到「第5條」相關內容。延遲 5–15 秒（medium reasoning）。

H-3 · audit log 寫進去 + 雜湊鏈完整

內網
docker exec ISO42001_rag_api python3 -c "
from rag_system.core.audit_logger import AuditLogger
from pathlib import Path
from datetime import datetime
f = Path(f'/app/data/audit_logs/audit_{datetime.now().strftime(\"%Y-%m-%d\")}.jsonl')
print(AuditLogger.verify_integrity(f))
"
預期
{'valid': True, 'total': N, 'broken_at': None, 'reason': 'chain intact'}
若 valid=False
看 broken_at 哪一行。若是第 1 行就是內網原本就有舊紀錄（雜湊鏈上線前），無解只能接受。若是中間斷掉就是新部署有問題，停下來查 log。

H-4 · 確認 query log 含新欄位（actions + model_response + prompt_version_hash）

內網
docker exec ISO42001_rag_api sh -c \
  'grep -h "deploy-check" /app/data/audit_logs/audit_*.jsonl | tail -1' \
  | python3 -c "
import json,sys
d=json.loads(sys.stdin.read())
for k in ['actions','model_response','prompt_version_hash','client_ip','prev_hash','entry_hash']:
    v=d.get(k,'(missing)')
    print(f'  {k}: {str(v)[:80]}')"
預期
六個欄位都有值。actions 應該長這樣： 
['classify=llm:legal', 'retrieve(docs=X,sources=Y)',
 'generate(citations=Z,tokens=N)', 'verify=passed(...)']

H-5 · 試做前清空資料（交付單位人員測試前）

此為試做：H 驗證跑過的測試查詢也會寫進 audit log。交付單位人員主動測試前， 先清空既有使用/測試資料，讓他們從乾淨狀態開始（使用資料與稽核日誌都是空的）。

內網
cd ~/ISO42001Deploy
./reset_data.sh        # 互動確認後執行
會清空 vs 會保留
清空：稽核日誌 audit_logs、對話歷史 conversations
保留：向量索引（知識庫）、法規語料、golden_dataset、版本快照（變更管理）
⚠ 只能在真實測試開始前執行一次
本腳本會重置稽核日誌的 SHA-256 雜湊鏈。單位人員真實測試一旦開始，稽核日誌即為 ISO 42001 A.6/A.9 證據，不可再執行本腳本。
