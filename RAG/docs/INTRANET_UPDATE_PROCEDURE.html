ISO42001 完整 Stack 更新流程

外部稽核準備乾淨基線 · rag-api / embed-proxy / jupyter / openwebui / keycloak / code-server / nginx
歷史資料遷移提醒：本頁原有備份步驟不包含 PostgreSQL、OpenWebUI、Keycloak、monitoring 與 admin 的完整 runtime，不能單獨用於跨主機歷史資料繼承。請改用 內網歷史資料遷移手冊與 backup_runtime.sh／restore_runtime.sh。
本文適用情境：從外部稽核準備乾淨基線建立或更新完整 ISO42001 stack。 目前專案已清空歷史稽核日誌、部署包、Docker tar、舊憑證與 .env。 全新啟動請先 cp .env.example .env 並填入強密碼與推論後端。
完整 stack：db（pgvector 基礎依賴）、embed-proxy、 rag-api、jupyter、openwebui、keycloak、 code-server、nginx。 nginx 對 host 使用 8088/8443，避開內網既有 80/443。

0更新包內容

檔案必要性說明
full-stack-code.zip★ 必要程式碼 + 文件（含 HTML）+ compose + .env.example；不含執行期資料與機密
images/*.tar★ 必要完整 stack images：rag-api、embed-proxy、jupyter、code-server、pgvector、openwebui、keycloak、nginx
MANIFEST.txt—檔案清單 + SHA-256 完整性驗證碼

1內網連線設定（本版重點）

模型連線由 .env 控制，且 embedding 走 embed-proxy（因 nv-embed-v2 為 Triton gRPC）：

項目內網值
LLM（gpt-oss-20b）http://10.53.100.12:7000/v1
Embedding（nv-embed-v2）Triton gRPC 10.53.100.12:9000（或 9001），由 embed-proxy 轉換
rag-api 取 embedding經 http://embed-proxy:7100/v1（內部 service）
請先確認 embed 的 gRPC port 是 9000 還是 9001，於 .env 設定 EMBED_GRPC_PORT（步驟 5）。compose 已改為由 .env 驅動， dev 與內網用各自 .env 切換，互不影響。

2前置備份（最重要，勿略過）

更新前務必備份。稽核日誌（雜湊鏈）一旦遺失無法重建，且為 ISO 42001 A.6/A.9 證據。

在內網部署目錄（如 ~/ISO42001Deploy）執行：

# 1. 用內建版本工具做快照 + tar.gz 備份（A.6.2.5 變更管理）
docker exec ISO42001_rag_api python3 scripts/version_tracker.py snapshot \
  -m "完整 stack 更新前備份" -o "<操作者>" -v "pre-full-stack-update" --backup

# 2. 另外手動備份執行期資料與 .env（雙保險）
ts=$(date +%Y%m%d_%H%M)
tar czf ~/iso42001_backup_${ts}.tar.gz \
  RAG/data/audit_logs RAG/data/versions RAG/data/processed .env
echo "備份完成：~/iso42001_backup_${ts}.tar.gz"

3傳輸與完整性驗證

# dev 端 → 內網／部署機
scp full-stack-code.zip images/*.tar MANIFEST.txt \
    <使用者>@10.53.100.12:~/iso42001-update/

# 內網端：核對 SHA-256 與 MANIFEST 一致
cd ~/iso42001-update
sha256sum full-stack-code.zip images/*.tar
#   ↑ 比對 MANIFEST.txt 內的值，必須完全相同

4載入 images

cd ~/iso42001-update
for f in images/*.tar; do docker load -i "$f"; done

5展開源碼 + 設定連線

安全：full-stack-code.zip 不含 audit_logs/、processed/、 versions/、.env，解壓不會覆蓋內網既有的稽核日誌、向量索引與機密。 
# 展開到部署目錄（覆蓋程式碼與文件，保留執行期資料）
cd ~/ISO42001Deploy
unzip -o ~/iso42001-update/full-stack-code.zip -d .

# 確認 .env 的連線值（沿用既有 .env，按本版內網端點調整）：
#   LLM_API_BASE=http://10.53.100.12:7000/v1
#   EMBED_API_BASE=http://embed-proxy:7100/v1
#   LLM_HOST=10.53.100.12
#   LLM_PORT=7000
#   EMBED_GRPC_PORT=9000        # 若實際為 9001 改此值
#   TRITON_MODEL_NAME=nv-embed-v2
nano .env     # 對照 .env.example 補上/調整以上欄位

6重建容器

cd ~/ISO42001Deploy
# 以新 image + 新程式碼重建完整 stack
docker compose up -d --build

# 等待健康
docker compose ps
也可執行 ./deploy.sh，腳本會自動偵測 UID/GID、產生自簽憑證、載入 images/*.tar 並啟動完整 stack。

7b疑難排解：embedding 回 500

已知議題：nv-embed-v2 為 Triton gRPC，embed-proxy 的 gRPC 埠若設錯（compose 預設 9000，實際常為 9001）、 或模型名/tensor 名不符，/v1/embeddings 會回 500，連帶 rag-api 檢索失敗。

embed-proxy 內建診斷端點 /ready，第一步先打它：

# 直接問 embed-proxy：Triton 連得到嗎？模型載入了嗎？期望什麼 tensor？
curl -s http://localhost:17100/ready | python3 -m json.tool
/ready 顯示研判處置
error: ...Connection refusedgRPC 埠/主機不通多半是埠：設 .env EMBED_GRPC_PORT=9001（或正確值），重啟 embed-proxy
server_live:true, model_ready:false模型未載入或名稱不符確認 Triton 上模型名＝TRITON_MODEL_NAME（nv-embed-v2）
hints: 名稱不符input/output tensor 名與 proxy 預期（query/documents/embeddings）不符對照 expected_inputs/outputs 調整 embed_proxy/proxy.py
hints: 相符連線與格式都對再送一筆實測 /v1/embeddings 確認
 
# 看 embed-proxy 即時錯誤日誌（本版已把錯誤寫入 stdout）
docker logs --tail 50 ISO42001_embed_proxy

# 改埠後重啟
nano .env        # EMBED_GRPC_PORT=9001
docker compose up -d --build embed-proxy
curl -s http://localhost:17100/ready | python3 -m json.tool   # 再確認

7c試做前清空資料（交付單位人員測試前）

此為試做：交付單位人員主動測試前，先清空既有的使用/測試資料，讓他們從乾淨狀態開始， 使用資料與稽核日誌都是空的。 
# 清空使用資料（保留法規語料、golden_dataset；外部稽核準備版本快照已清空）
./reset_data.sh        # 互動確認後執行；會 TRUNCATE conversations、清 audit_logs
會清空會保留
稽核日誌（audit_logs）向量索引（pgvector 知識庫）
對話歷史（conversations 表）法規語料 converted_md
版本 snapshot（外部稽核準備已清空）必要時重新產生版本快照
本腳本會重置稽核日誌的雜湊鏈，僅限「真實測試開始前」執行一次。 單位人員真實測試一旦開始，稽核日誌即為 ISO 42001 A.6/A.9 證據，不可再清空。

# 4. 確認查詢正常、稽核日誌完整
